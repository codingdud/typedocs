Basic

Basic Types

Definition: TypeScript extends JavaScript with a type system. Basic types are the fundamental building blocks used to annotate variables, parameters, and return values. They enable compile-time checks that prevent common runtime bugs.

let isDone: boolean = false;
let age: number = 25;           // int, float, hex, binary
let name: string = "Alice";
let list: number[] = [1, 2, 3];  // or Array<number>
let x: [string, number] = ["hi", 1]; // Tuple

let notSure: any = 4;     // opt-out of type checking
let unknown: unknown = 4;  // safer than any
let n: null = null;
let u: undefined = undefined;
let sym: symbol = Symbol("key");
function log(): void {}  // no return value
function fail(): never { throw new Error(); } // never returns
booleannumberstringarraytupleanyunknownnullundefinedvoidneversymbol
Interview Tip

Be ready to explain: any vs unknown — unknown is the type-safe counterpart; you must narrow it before use. void vs never — void means "returns nothing useful", never means "never completes".

Basic

Variables

Definition: TypeScript supports let, const, and var with optional type annotations. TypeScript can also infer types automatically — if you assign a value, TypeScript derives the type without an explicit annotation.

// Explicit annotation
let score: number = 100;

// Type inference — TypeScript infers number
let message = "hello";  // inferred as string

// const infers a literal type
const status = "active"; // type: "active" not string

// Union type for flexibility
let id: string | number = "abc";
id = 42; // also valid

// Type assertion (when you know better than TS)
let input = getInput() as string;

// Non-null assertion (use sparingly)
let el = document.getElementById("app")!;
Interview Tip

Interviewers love: type inference vs annotation — prefer inference when possible, annotate for clarity on function signatures. const narrowing — const x = "open" gives literal type "open", useful for discriminated unions.

Basic

Const & Literal Types — const vs let Widening

Core rule: When you declare with const, TypeScript infers the narrowest possible type — the exact literal value. When you declare with let, TypeScript widens to the general primitive type because the value could be reassigned.

// ── STRING LITERALS ──────────────────────────────
const name1 = "Alice";   // type: "Alice"  ← literal, not string
let   name2 = "Alice";   // type: string   ← widened

// ── NUMBER LITERALS ──────────────────────────────
const one  = 1;          // type: 1        ← literal, not number
let   num  = 1;          // type: number   ← widened

// ── BOOLEAN LITERALS ─────────────────────────────
const yes  = true;       // type: true     ← literal, not boolean
let   flag = true;       // type: boolean  ← widened

// ── WHY IT MATTERS ───────────────────────────────
declare function move(dir: "left" | "right"): void;

const d1 = "left";     move(d1); // ✅ type is "left"
let   d2 = "left";     move(d2); // ❌ type is string — too wide

// Fix for let: explicit literal annotation
let d3: "left" | "right" = "left";
move(d3); // ✅

const — literal type

  • Value can never change → safest narrowing
  • Type = exact value: "Alice", 1, true
  • Works directly with union params
  • Used in discriminated unions as discriminants

let — widened type

  • Value can change → TypeScript infers broadly
  • Type = primitive: string, number, boolean
  • Too wide for literal union params
  • Fix: annotate explicitly or use as const
Interview Tip

"Why is let x = 'left' not assignable to 'left' | 'right'?" — because let widens to string. TypeScript does this because let signals the value might change. const proves it won't, so the literal type is safe.

Basic

as const — Freeze the Entire Shape

as const is a type assertion that tells TypeScript: "treat every value in this expression as its narrowest possible literal type and mark everything readonly." It works on primitives, object literals, and arrays.

// ── ON A PRIMITIVE (equivalent to const variable) ─
let x = "open" as const;  // type: "open"  (even on let!)

// ── ON AN OBJECT ──────────────────────────────────
const config = {
  host: "localhost",
  port: 3000,
  env:  "production",
};
// Without as const: { host: string; port: number; env: string }

const config2 = {
  host: "localhost",
  port: 3000,
  env:  "production",
} as const;
// With as const:
// { readonly host: "localhost"; readonly port: 3000; readonly env: "production" }
//    ↑ ALL values are literal types  ↑ ALL props are readonly

// ── ON AN ARRAY ───────────────────────────────────
const colors  = ["red", "green", "blue"];
// type: string[]  ← widened array

const colors2 = ["red", "green", "blue"] as const;
// type: readonly ["red", "green", "blue"]  ← literal tuple

// ── ON A FUNCTION ARGUMENT ────────────────────────
declare function setEnv(e: "dev" | "prod"): void;

const opts = { env: "prod" };          setEnv(opts.env); // ❌ string
const opts2 = { env: "prod" } as const; setEnv(opts2.env); // ✅ "prod"
Watch Out

as const only affects the TypeScript type — the compiled JavaScript is identical. The object is not actually frozen at runtime (use Object.freeze() for that). Also: as const is deep — nested objects also become readonly literal types.

Advanced

Literal Types in Depth — The "name" Type & 1 Type

A literal type is a type that represents exactly one specific value. TypeScript has string literals, number literals, boolean literals, and (via as const) bigint and symbol literals.

// ── STRING LITERAL TYPE ───────────────────────────
type Name = "Alice";   // only "Alice" is assignable
let n: Name = "Alice";  // ✅
// let n2: Name = "Bob"; // ❌

// ── NUMBER LITERAL TYPE ───────────────────────────
type One = 1;            // only the number 1
type DiceRoll = 1 | 2 | 3 | 4 | 5 | 6;

function roll(): DiceRoll { return (Math.floor(Math.random() * 6) + 1) as DiceRoll; }

// ── BOOLEAN LITERAL TYPE ──────────────────────────
type AlwaysTrue = true;  // only true (not false)

// ── LITERAL UNION — the real power ───────────────
type Direction = "north" | "south" | "east" | "west";
type StatusCode = 200 | 201 | 400 | 401 | 404 | 500;
type Toggle     = true | false;  // same as boolean

// ── RETURN TYPE LITERAL ───────────────────────────
function compare(a: number, b: number): -1 | 0 | 1 {
  return a < b ? -1 : a > b ? 1 : 0;
}
const r = compare(2, 3);  // type: -1 | 0 | 1  (not number)

// ── WIDENING TRAP ON OBJECT PROPERTY ─────────────
const obj = { status: "ok" };       // status: string  ← widened!
const obj2 = { status: "ok" } as const; // status: "ok" ← literal

type ObjStatus  = typeof obj.status;   // string
type ObjStatus2 = typeof obj2.status;  // "ok"
Common Misunderstanding

type Name = "Alice" is not a value — it is a type that only permits the string "Alice". People often confuse this with a string variable. Similarly type One = 1 is the type of the number one, not the number itself. These are most useful combined into unions: "left" | "right", 1 | 2 | 3.

Advanced

satisfies Operator & Template Literal Types with const

satisfies (TS 4.9) validates a value against a type without widening it — you keep the literal inference while still catching mistakes. Template literal types let you build new string literal types by interpolating others.

// ── satisfies — validate shape, keep literals ─────
type Palette = { [k: string]: [number, number, number] | string };

const colors = {
  red:   [255, 0, 0],
  green: "#00ff00",
} satisfies Palette;

// Without satisfies: you'd annotate as Palette and lose literal info
colors.red;    // ✅ type: [255, 0, 0]  — literal tuple kept!
colors.green;  // ✅ type: "#00ff00"   — string literal kept!
// colors.blue; // ❌ compile error — not in Palette check

// Without satisfies (annotation widens):
// const colors: Palette = { ... }
// colors.red   → [number,number,number] | string  ← widened, loses tuple

// ── template literal types with const ─────────────
const prefix = "get" as const;  // type: "get"
type Prefix  = typeof prefix;    // "get"

type Keys    = "Name" | "Age" | "Email";
type Getters = `${Prefix}${Keys}`;
// → "getName" | "getAge" | "getEmail"

// Auto-generating getter types from an object:
interface Person { name: string; age: number; }
type PersonGetters = {
  [K in keyof Person as `get${Capitalize<string & K>}`]: () => Person[K];
};
// → { getName: () => string; getAge: () => number }

Use satisfies when

  • You want shape validation and literal inference
  • The annotation would widen your types
  • You need autocomplete on specific keys

Use annotation (: Type) when

  • You want to widen intentionally
  • The type is the public API contract
  • You need to restrict what can be assigned
Interview Tip

satisfies answers the question: "How do I validate the shape of an object without losing its precise type?" The classic example is a config object — you want to ensure all required keys exist, but also keep port typed as 3000 (not number) for exhaustive switch branches.

Watch Out

Most Misunderstood const / Literal Scenarios

These are the exact patterns that trip up TypeScript developers in interviews and code reviews.

// ── MISUNDERSTANDING 1: const object DOES NOT preserve literals ─
const user = { role: "admin" };
// user.role: string  ← NOT "admin"!
// const only makes the binding (user) non-reassignable
// the object's properties are still mutable → TypeScript widens them

const user2 = { role: "admin" } as const;
// user2.role: "admin"  ← literal ✅

// ── MISUNDERSTANDING 2: function param doesn't narrow ───────────
function getRole() { return "admin"; }
// return type: string  ← widened (function body is not const)

function getRole2(): "admin" { return "admin"; }
// return type: "admin" ← explicit annotation needed

function getRole3() { return "admin" as const; }
// return type: "admin" ← as const on return value also works

// ── MISUNDERSTANDING 3: array item type ─────────────────────────
const roles  = ["admin", "user"];       // string[]  ← widened
const roles2 = ["admin", "user"] as const; // readonly ["admin","user"]

type Role  = typeof roles[number];   // string
type Role2 = typeof roles2[number];  // "admin" | "user" ✅

// ── MISUNDERSTANDING 4: spreading loses as const ────────────────
const base = { x: 1, y: 2 } as const;
const copy = { ...base };    // { x: number; y: number } ← literals lost!
const copy2 = { ...base } as const; // { readonly x: 1; readonly y: 2 } ✅

// ── MISUNDERSTANDING 5: typeof on const gives literal ───────────
const MAX = 100;
type Max = typeof MAX;  // type: 100  (not number)
let   MAX2 = 100;
type Max2 = typeof MAX2; // type: number  (widened)
Interview Tip

The single most common interview gotcha: const obj = { role: "admin" } does NOT make role a literal type. const only prevents rebinding the variable — it says nothing about mutability of the object's properties. You need as const (or Object.freeze) to get literal property types.

Basic

Function Types — What They Are & 3 Ways to Write Them

A function type describes the shape of a function — its parameter types and return type — without providing an implementation. You use them to type variables, parameters, and properties that hold functions.

// ── SYNTAX 1: arrow style (most common) ──────────────
type Add = (a: number, b: number) => number;

// Use it to type a variable that holds a function
const add: Add = (a, b) => a + b;   // params inferred as number

// Use it to type a callback parameter
function runTwice(fn: Add): number {
  return fn(1, 2) + fn(3, 4);
}
runTwice((a, b) => a + b); // ✅ a, b inferred as number (contextual typing)

// ── SYNTAX 2: inline in a function parameter ──────────
function transform(
  value: string,
  fn: (input: string) => string   // ← inline function type
): string {
  return fn(value);
}
transform("hello", s => s.toUpperCase()); // ✅

// ── SYNTAX 3: object method shorthand ─────────────────
type Validator = {
  validate(input: string): boolean;  // method shorthand
  format: (s: string) => string;    // property arrow syntax
};
Interview Tip

Contextual typing — when you pass a lambda to a typed parameter, TypeScript infers the parameter types from the function type. You don't need to annotate them again. This is why arr.forEach(s => s.toUpperCase()) works without annotating s.

Basic

Parameters — Optional, Default & Rest

// ── REQUIRED ─────────────────────────────────────────
function greet(name: string): string {
  return `Hello, ${name}`;
}

// ── OPTIONAL  (?) ────────────────────────────────────
// optional param has type  string | undefined  inside the function
function greet2(name: string, title?: string): string {
  return title ? `${title} ${name}` : name;
}
greet2("Alice");           // ✅ title = undefined
greet2("Alice", "Dr");    // ✅

// ── DEFAULT ──────────────────────────────────────────
// default param is NOT undefined inside — it has the default value
function connect(host: string, port: number = 3000): string {
  return `${host}:${port}`;
}
connect("localhost");       // ✅ port = 3000
connect("localhost", 8080); // ✅ port = 8080

// ── REST (variadic) ───────────────────────────────────
// rest param must be last and is always an array type
function sum(first: number, ...rest: number[]): number {
  return rest.reduce((acc, n) => acc + n, first);
}
sum(1, 2, 3, 4); // first=1, rest=[2,3,4] → 10

optional ?

  • Type inside: string | undefined
  • Caller may omit it
  • No default value substituted
  • Must be last (or after other optionals)

default = val

  • Type inside: string (never undefined)
  • Caller may omit it OR pass undefined
  • Default value substituted when absent
  • Can appear anywhere (unlike optional)
Advanced

Call Signatures & Constructor Signatures

Some values in JavaScript are callable (like functions) but also have properties. And some can be called with or without new. TypeScript models both with call signatures and constructor signatures inside object types.

Call signature — function that also has properties

// Problem: you need a function that also carries a .description property
// You cannot do: type F = (arg: number) => boolean & { des: string }
// Solution: call signature inside an object type

type DescFunction = {
  des: string;              // regular property
  (arg: number): boolean;  // ← call signature (no "function" keyword)
};

function getDesc(fn: DescFunction) {
  console.log(fn.des);   // access property
  return fn(123);        // call it as a function
}

// Build a value that satisfies DescFunction:
function myFun(arg: number): boolean { return arg > 0; }
myFun.des = "checks positive";  // attach the property
getDesc(myFun);                   // ✅

Constructor signature — typed new

// Type that can ONLY be used with new:
type Newable = { new(s: string): object };

// Type that can be called BOTH ways (like Date):
type CallOrConstruct = {
  new(s: string): object;  // new CallOrConstruct("hi") → object
  (n: number): number;     // CallOrConstruct(42) → number
};

// Real-world example — factory function accepting a class:
function create<T>(ctor: { new(): T }): T {
  return new ctor();
}
class Dog { bark() { return "woof"; } }
const dog = create(Dog);   // type: Dog ✅
Interview Tip

Call signatures explain how React.FC, express.Router, and other library types work — they are callable objects with extra properties. Constructor signatures are how Angular/NestJS dependency injection types (Type<T>) are modelled.

Advanced

void Return Type — The Surprising Rule

void means "the caller should not use the return value", but it does not mean the function is forced to return nothing. There are two different contexts and the rule differs between them.

// ── CONTEXT 1: explicit void annotation on a declaration ─────
// → function MUST return nothing (or undefined)
function log(msg: string): void {
  console.log(msg);
  // return 42; // ❌ Error: Type 'number' is not assignable to type 'void'
}

// ── CONTEXT 2: void in a function TYPE (callback) ─────────────
// → function MAY return any value, but callers must IGNORE it
type VoidCallback = () => void;

const cb1: VoidCallback = () => true;   // ✅ returns boolean — allowed!
const cb2: VoidCallback = () => 42;     // ✅ returns number  — allowed!
const cb3: VoidCallback = () => {};     // ✅ returns nothing — allowed!

// Real reason this exists — Array methods like forEach:
// forEach signature: (fn: (value: T) => void) => void
// This lets you pass any function, even one that returns something:
[1, 2, 3].forEach(n => n * 2);  // ✅ return value is discarded, no error

// ── USING THE RETURN VALUE IS STILL DISALLOWED ────────────────
function run(fn: () => void) {
  const result = fn();
  // result: void — you cannot use it meaningfully
  // result.toString(); // ❌ Error
}

// ── void vs never ─────────────────────────────────────────────
function noReturn(): void  { }              // returns, just no value
function crashes():  never { throw new Error(); }  // NEVER returns
function loops():    never { while(true){} }       // NEVER returns
Common Mistake

People think void always means "returns nothing". It only enforces that on function declarations with an explicit : void annotation. On a function type like () => void, implementations can return a value — it will just be ignored by the caller. This is why [].push (returns number) can be passed to a forEach callback typed as () => void.

Advanced

Generic Function Types

A generic function type is a function type where one or more parameter or return types are type variables — resolved by the caller. The type variable links the input and output so TypeScript can track the relationship.

// ── BASIC GENERIC FUNCTION ────────────────────────────
// Without generics: you'd use any and lose type information
function identity<T>(value: T): T { return value; }

identity("hello");  // T inferred as string → returns string
identity(42);       // T inferred as number → returns number
identity<boolean>(true); // T explicitly set

// ── GENERIC CALLBACK — linking input and output ───────
function map<Input, Output>(
  arr: Input[],
  fn:  (item: Input) => Output
): Output[] {
  return arr.map(fn);
}
const nums = map(["1", "2"], n => parseInt(n));
// Input=string, Output=number → nums: number[]

// ── CONSTRAINT — only accept types with .length ───────
function longest<T extends { length: number }>(a: T, b: T): T {
  return a.length >= b.length ? a : b;
}
longest("alice", "bob");       // ✅ string has .length
longest([1, 2], [3, 4, 5]);  // ✅ array has .length
// longest(10, 20);           // ❌ number has no .length

// ── SPECIFYING TYPE ARGS when inference fails ──────────
function combine<T>(a: T[], b: T[]): T[] { return [...a, ...b]; }
// combine([1], ["hello"]);                  // ❌ T can't be both
combine<number | string>([1], ["hello"]); // ✅ explicit T

// ── GENERIC TYPE ALIAS for function types ─────────────
type Reducer<S, A> = (state: S, action: A) => S;

const countReducer: Reducer<number, "inc" | "dec"> =
  (state, action) => action === "inc" ? state + 1 : state - 1;
Interview Tip

Generic vs any: any discards the type — you lose the relationship between input and output. Generics preserve it. identity<string> guarantees the return is a string; identity: any returns any. The constraint T extends X is how you say "T must have at least these properties".

Advanced

Function Overloads

Overloads let a single function accept different combinations of argument types and return different types depending on what was passed. You write overload signatures (the public API) then one implementation signature (the body) that must be broad enough to handle all cases.

// ── ANATOMY ───────────────────────────────────────────
function makeDate(timestamp: number): Date;              // ← overload 1
function makeDate(month: number, day: number, year: number): Date; // ← overload 2
function makeDate(mOrTs: number, d?: number, y?: number): Date { // ← implementation
  if (d !== undefined && y !== undefined) {
    return new Date(y, mOrTs - 1, d);
  }
  return new Date(mOrTs);
}
makeDate(1716000000000);   // ✅ matches overload 1
makeDate(5, 18, 2024);     // ✅ matches overload 2
// makeDate(5, 18);         // ❌ no overload matches 2 args

// ── RULE: implementation must be compatible with ALL overloads ──
// The implementation signature is NOT visible to callers —
// callers only see the overload signatures above it.

// ── OVERLOAD for different input/output types ──────────
function process(x: string): string[];
function process(x: number): number;
function process(x: string | number): string[] | number {
  return typeof x === "string" ? x.split("") : x * 2;
}
const a = process("hi");  // a: string[]  ← TypeScript picks overload 1
const b = process(5);    // b: number   ← TypeScript picks overload 2

// ── SPREAD TUPLE TRICK (avoid an extra overload) ───────
const args = [8, 5] as const;         // readonly [8, 5]
const angle = Math.atan2(...args);    // ✅ TS knows it's exactly 2 numbers
// without as const: Math.atan2 would error — spread of number[] is not allowed
Watch Out

Two overload traps: ① The implementation signature must be broader than all overloads — (x: string | number) not just (x: string). ② Overloads are matched top-to-bottom — put the most specific overload first. The implementation signature is invisible to callers so it never matches directly.

Classes

Definition: TypeScript classes are blueprints for objects. They support constructors, properties, methods, access modifiers, inheritance via extends, and interface implementation via implements. TypeScript adds compile-time OOP safety on top of ES6 classes.

class Animal {
  private name: string;
  protected species: string;
  readonly id: number;

  // Constructor shorthand (declares + assigns in one go)
  constructor(public nickname: string, name: string) {
    this.name = name;
    this.id = Math.random();
  }

  speak(): string { return `I am ${this.name}`; }
}

class Dog extends Animal {
  speak(): string { return "Woof!"; } // override
}

// Abstract class
abstract class Shape {
  abstract area(): number;  // must be implemented
  describe() { return "I am a shape"; }
}
Interview Tip

Highlight constructor shorthand — constructor(public name: string) simultaneously declares and assigns a property. Also explain abstract class vs interface — abstract classes can have implementation logic; interfaces cannot.

Basic

Constructor Shorthand — No Top-Level Declaration Needed

Normally you must declare a property at the top of the class AND assign it inside the constructor — two steps. A constructor shorthand (also called a parameter property) collapses both into a single constructor parameter by adding an access modifier to it.

Without shorthand — 2 steps

class User {
  // Step 1 — declare at top level
  name: string;
  age:  number;
  private role: string;

  // Step 2 — assign in constructor
  constructor(
    name: string,
    age:  number,
    role: string
  ) {
    this.name = name;
    this.age  = age;
    this.role = role;
  }
}

With shorthand — 1 step ✅

class User {
  // No top-level declarations needed!
  // The modifier in the param does it all.


  constructor(
    public  name: string,
    public  age:  number,
    private role: string
  ) {
    // nothing to write here —
    // TypeScript does it automatically
  }
}
// Both produce IDENTICAL compiled JavaScript:
// class User {
//   constructor(name, age, role) {
//     this.name = name;
//     this.age  = age;
//     this.role = role;
//   }
// }

// ── THE RULE: a modifier on the param = shorthand ────
// public / private / protected / readonly → shorthand
// no modifier → plain parameter, NOT stored on the class

class Point {
  constructor(
    public readonly x: number,   // shorthand — public + readonly
    public readonly y: number,   // shorthand — public + readonly
    label: string                 // ← NO modifier → NOT a property
  ) {}                            //   label is only available during
}                                 //   constructor execution

const p = new Point(1, 2, "origin");
p.x;      // ✅ 1
p.y;      // ✅ 2
// p.label  ❌ Property 'label' does not exist on type 'Point'

// ── MIXING: shorthand + extra constructor logic ───────
class Circle {
  area: number;   // ← still declared at top (computed, not a param)

  constructor(public readonly radius: number) {
    // shorthand handled 'radius'; manually compute 'area'
    this.area = Math.PI * radius ** 2;
  }
}
const c = new Circle(5);
c.radius; // ✅ 5
c.area;   // ✅ 78.53...
Common Mistake

Forgetting the modifier on a constructor parameter and wondering why this.name doesn't exist. The parameter name: string and the shorthand public name: string look almost identical but behave completely differently — without a modifier it is just a local variable that disappears after the constructor runs.

Advanced

Abstract Class vs Interface — When to Use Which

Both define a contract that other classes must follow, but they differ fundamentally in what they can contain and how they are used at runtime.

Abstract Class

abstract class Animal {
  // ✅ can have concrete fields
  name: string;

  // ✅ can have a constructor
  constructor(name: string) {
    this.name = name;
  }

  // ✅ concrete method — shared logic
  breathe() {
    return `${this.name} breathes`;
  }

  // ✅ abstract method — subclass MUST implement
  abstract speak(): string;
}

class Dog extends Animal {
  speak() { return "Woof"; }
}

// new Animal() ❌ cannot instantiate
new Dog("Rex").breathe(); // ✅

Interface

interface Animal {
  // ✅ property shapes
  name: string;



  // ❌ no constructor allowed




  // ❌ no implementation — only signature
  breathe(): string;

  // all methods are "abstract" by nature
  speak(): string;
}

class Dog implements Animal {
  name = "Rex";
  breathe() { return `${this.name} breathes`; }
  speak()   { return "Woof"; }
}

// erased at runtime — no JS output

interface Serializable {
  toJSON(): string;
}

abstract class BaseEntity implements Serializable {
  constructor(public readonly id: string) {}

  // concrete shared logic
  toJSON(): string { return JSON.stringify({ id: this.id }); }

  // subclass must implement their own label
  abstract label(): string;
}

class User extends BaseEntity {
  constructor(id: string, public name: string) { super(id); }
  label() { return `User: ${this.name}`; }
}

u instanceof User;     // ✅ true
u instanceof BaseEntity; // ✅ true (abstract class exists at runtime)
Interview Tip

Decision rule: need shared implementation or a constructor? → abstract class. Need a pure contract that multiple unrelated classes can satisfy, or need multiple inheritance? → interface. In practice you often use both: the abstract class implements the interface for shared logic, and subclasses extend the abstract class.

Basic

Interfaces

Definition: An interface defines the shape of an object — a contract that specifies what properties and methods an object must have. TypeScript uses structural typing, meaning any object that satisfies the shape is valid, regardless of its class name.

interface User {
  id: number;
  name: string;
  email?: string;           // optional property
  readonly createdAt: Date;  // cannot reassign
  greet(): string;           // method signature
}

// Extending interfaces
interface Admin extends User {
  role: string;
}

// Declaration merging (unique to interfaces)
interface Window { myCustomProp: string; }
interface Window { anotherProp: number; }
// Both declarations are merged automatically

// Implementing an interface in a class
class UserImpl implements User {
  id = 1; name = "Bob"; createdAt = new Date();
  greet() { return `Hi, I'm ${this.name}`; }
}
Interview Tip

Structural typing means: { id: 1, name: 'x', greet() {...} } satisfies User even without class or implements. Also mention declaration merging — unique to interfaces, often used to augment third-party types like Window or Express.Request.

Declaration Merging

Declaration Merging — interfaces Only

Declaring the same interface name twice merges the two declarations into one shape. TypeScript requires all properties from both blocks. type does not support this — a duplicate type name is always an error.

interface — merges 
interface Window {
  title: string;
}
interface Window {
  analytics: Analytics;
}
// Result: { title, analytics }
// window.analytics now type-safe ✅
type — error 
type Window = {
  title: string;
};
type Window = {        // ❌
  analytics: Analytics;
};
// Error: Duplicate identifier 'Window'

// Real use: adding types to a third-party library
// In your project's global.d.ts:
declare global {
  interface Window {
    myPlugin: () => void;
  }
  interface Array<T> {
    last(): T | undefined;
  }
}
// Now window.myPlugin and [].last() are type-safe everywhere
Gotcha

Merging only works if the types of shared properties are compatible. If two declarations define the same property with conflicting types, TypeScript errors. For methods, merging creates an overload — the most specific overload should be declared last.

Composition

extends vs & — How Composition Differs

Both combine shapes, but they handle property conflicts differently — this is the most important practical distinction.

interface extends — conflict is an error 
interface A { id: number; }
interface B { id: string; }

interface C extends A, B {} // ❌
// Error: 'id' types are incompatible
// TypeScript stops you at the definition
type & — conflict becomes never 
type A = { id: number; };
type B = { id: string; };

type C = A & B;  // no error yet
// C.id is type: never
// Error surfaces only at use site

// interface extends can mix interface + type aliases
type Named  = { name: string };
type Aged   = { age: number };
interface Person extends Named, Aged {
  email: string;
}
// Person = { name, age, email } ✅

// & composes freely across types and interfaces
type Admin = Person & { role: string; permissions: string[] };

// extends is stricter (catches conflicts early) — prefer it for OOP hierarchies
// & is more flexible (lazy evaluation) — prefer it for utility/computed types
Interview Tip

extends is an eager compatibility check — conflicts are caught at the extends declaration. & is lazy — conflicts resolve to never and only blow up when you try to assign a value. Prefer extends in inheritance hierarchies precisely because it fails early.

Advanced

implements · Mapped Types · Conditional Types

implements — a class can implement both an interface and an object-shaped type. The only restriction is that type must represent an object shape (not a union or primitive).

interface Serializable { toJSON(): string; }
type      Loggable     = { log():    void;   };

class Event implements Serializable, Loggable {   // ✅ both work
  toJSON() { return "{}"; }
  log()    { console.log(this); }
}

type Union = Cat | Dog;
// class X implements Union {}  // ❌ cannot implement a union type

Mapped types — only type supports them directly. interface cannot contain [K in ...] syntax.

// type — supports mapped types ✅
type Readonly<T> = { readonly [K in keyof T]: T[K] };
type Partial<T>  = { [K in keyof T]?: T[K] };
type Flags = { [K in "read" | "write" | "exec"]: boolean };

// interface — ❌ cannot do this
// interface Flags { [K in "read"|"write"]: boolean }  // syntax error

Conditional types — exclusive to type. They let the output type depend on a type-level condition.

type IsString<T> = T extends string ? "yes" : "no";
type A = IsString<string>;  // "yes"
type B = IsString<number>;  // "no"

type Unwrap<T> = T extends Promise<infer U> ? U : T;
type R = Unwrap<Promise<number>>; // number
Interview Tip

All of TypeScript's built-in utility types (Partial, Required, Readonly, Pick, Omit, ReturnType…) are implemented as type aliases using mapped and conditional types — they cannot be expressed as interfaces.

Performance & Decision

Performance Implications & When to Choose

Compiler performance — interfaces are named and cached by the TypeScript compiler. When the same interface is referenced many times, the checker reuses the cached result. Complex type intersections (A & B & C & D) are expanded inline at each use site, which can slow large codebases.

// Prefer interface for shapes referenced many times across the codebase
interface User { id: number; name: string; email: string; }

// Deep intersection chains — slower (each use re-expands)
type FullUser = Base & Auth & Profile & Settings;

// Better: lift into an interface once — cached after first check
interface FullUser extends Base, Auth, Profile, Settings {}

Decision guide — choose based on what you actually need:

Augmenting a third-party typeinterfaceOnly interfaces support module augmentation
Union of typestypeinterface cannot express unions

Tuple or primitive aliastypeinterface cannot alias non-objects

Interview Tip

The TypeScript team's own style guide says: "Use interface when you can, type when you must." In practice: if it's a shape that a class will implement or a library consumer will extend → interface. If it's a union, a tuple, a computed/conditional type, or a utility → type. Both are erased at runtime — the choice is entirely about authoring ergonomics and tooling behaviour.

Advanced

Static Properties & Methods

static members belong to the class itself, not to any instance. They are accessed via the class name, not via this on an object. Static members are shared across all instances.

class MathHelper {
  // static property — one shared value, on the class
  static PI: number = 3.14159;

  // static method — no instance needed
  static circleArea(r: number): number {
    return MathHelper.PI * r * r; // access via class name, not this
  }
}

MathHelper.PI;              // ✅ 3.14159
MathHelper.circleArea(5); // ✅ 78.53...

const m = new MathHelper();
// m.PI           ❌ Error: static member not on instance
// m.circleArea() ❌ same

// ── STATIC + ACCESS MODIFIERS ────────────────────────
class IdGenerator {
  private static nextId: number = 1; // private + static

  static generate(): number {
    return IdGenerator.nextId++;     // only accessible inside class
  }
}

console.log(IdGenerator.generate()); // 1
console.log(IdGenerator.generate()); // 2
// IdGenerator.nextId  ❌ private

// ── STATIC + READONLY ────────────────────────────────
class Config {
  static readonly MAX_RETRIES = 3;   // can never be changed
  static readonly BASE_URL    = "https://api.example.com";
}

// ── INHERITING STATIC MEMBERS ────────────────────────
class Animal {
  static kingdom = "Animalia";
  static describe() { return `Kingdom: ${this.kingdom}`; }
}
class Dog extends Animal {
  static kingdom = "Canidae";   // overrides parent static
}
console.log(Dog.describe()); // "Kingdom: Canidae" (this = Dog)
Watch Out

Inside a static method, this refers to the class (constructor function), not an instance. You cannot access instance properties (this.name) from a static method. Access other static members via the class name or via this (which works when inherited, as shown above).

Advanced

Static Blocks — Initialization Logic for Static Members

A static block (static { ... }) runs once when the class is first loaded. It lets you run complex initialization logic for static fields that can't be expressed in a simple assignment — like try/catch, loops, or reading from environment.

// ── BASIC: static block runs once at class load ───────
class AppConfig {
  static debug: boolean;
  static maxRetries: number;

  static {
    // Runs once when AppConfig is first referenced
    AppConfig.debug      = process.env.NODE_ENV !== "production";
    AppConfig.maxRetries = AppConfig.debug ? 1 : 3;
    console.log("AppConfig initialized");
  }
}
// "AppConfig initialized" logged exactly once
console.log(AppConfig.debug);       // true or false
console.log(AppConfig.maxRetries);  // 1 or 3

// ── USE CASE: initialization that can fail ────────────
class DatabasePool {
  static url: string;

  static {
    try {
      DatabasePool.url = requireEnv("DATABASE_URL");
    } catch {
      DatabasePool.url = "sqlite://fallback.db";
    }
  }
}

// ── PRIVATE STATIC + STATIC BLOCK together ───────────
class Counter {
  static #count = 0;   // private static field

  static {
    // Can access private # fields inside static block of same class
    Counter.#count = 1000; // seed starting value
    console.log("Counter seeded at", Counter.#count);
  }

  static next() { return ++Counter.#count; }
  static reset() { Counter.#count = 1000; }
}

console.log(Counter.next());  // 1001
console.log(Counter.next());  // 1002
Counter.reset();
console.log(Counter.next());  // 1001

// ── MULTIPLE STATIC BLOCKS (run in order) ────────────
class Ordered {
  static log: string[] = [];
  static { Ordered.log.push("first");  }
  static { Ordered.log.push("second"); }
}
console.log(Ordered.log); // ["first", "second"]

static block CAN

  • Run try/catch during initialization
  • Read environment variables / config
  • Access private # fields of the class
  • Run multiple blocks (executed top-to-bottom)
  • Call other static methods of the class

static block CANNOT

  • Use return / await directly
  • Access instance properties (this.x)
  • Run more than once (it's one-shot)
  • Be called manually — runs automatically
Interview Tip

Static blocks solve the problem of "static fields that need complex initialization". Before static blocks, you had to call a static method after the class definition (MyClass.init()). Now the initialization is part of the class body, guaranteed to run before any external code touches the class.

Overview

Decorators — Legacy (experimentalDecorators)

What: A decorator is a function attached to a class, method, accessor, property, or parameter via @expression. It runs at class definition time (not at instantiation). Enable with "experimentalDecorators": true in tsconfig.

Two forms for every decorator type:

// 1. Direct — @FnName  (no parentheses)
//    The function itself IS the decorator
@Sealed
class App {}

// 2. Factory — @FnName()  (with parentheses)
//    A function that RETURNS the decorator (allows passing config)
@Entity("users")
class User {}

Evaluation order with multiple decorators: factories run top → bottom, resulting decorators apply bottom → top (like function composition f(g(x))).

Class Method Accessor Property Parameter
Note

This section covers the legacy experimental decorator API (TS 4.x / Angular / NestJS). TypeScript 5.0 introduced Stage 3 decorators with a different signature — both exist today, but most production code still uses the experimental API.

Class Decorator

Class Decorator — (constructor: Function) => void

Receives the class constructor. Can seal it, add properties to prototype, or replace the constructor entirely.

Direct form — @Sealed

function Sealed(constructor: Function) {
  Object.seal(constructor);
  Object.seal(constructor.prototype);
}

@Sealed
class App { version = "1.0"; }

Factory form — @Entity("users")

function Entity(tableName: string) {
  return function(constructor: Function) {
    constructor.prototype.__tableName = tableName;
  };
}

@Entity("users")
class User { name = "Animesh"; }
// (new User() as any).__tableName → "users"

Why use a factory? — When you need to pass configuration (table name, route path, feature flag) into the decorator. The outer function captures the config, the inner function receives the constructor.

Use Cases

Angular: @Component({ selector, template }) · NestJS: @Controller("/api") · TypeORM: @Entity("users") · Common patterns: sealing classes, adding metadata, logging instantiation, making classes singleton.

Method Decorator

Method Decorator — (target, key, descriptor) => void

Receives target (prototype), propertyKey (method name), descriptor (PropertyDescriptor). The key pattern is function delegation — save the original method, replace descriptor.value with a wrapper that calls the original.

Direct form — @LogExecution

function LogExecution(target: any, key: string, desc: PropertyDescriptor) {
  const original = desc.value;             // save original
  desc.value = function(...args: any[]) {
    console.log(`→ ${key}`, args);       // before
    return original.apply(this, args); // delegate to original
  };
}

Factory form — @AddStar()

function Prefix(symbol: string) {
  return function(target: any, key: string, desc: PropertyDescriptor) {
    const fn = desc.value;
    desc.value = function(...args: any[]) {
      return `${symbol}${fn.apply(this, args)}${symbol}`;
    };
  };
}

class Greeter {
  @Prefix("*")
  greet() { return "hello"; }
}
// new Greeter().greet() → "*hello*"

Why function delegation? — You want to wrap behavior (logging, timing, caching, auth checks) without modifying the original method body. The decorator adds cross-cutting logic while original.apply(this, args) preserves this context and passes through all arguments untouched.

Use Cases

NestJS: @Get("/path"), @UseGuards() · Common patterns: logging, execution timing, caching/memoization, retry logic, auth/permission checks, debounce/throttle, validation of return values.

Accessor Decorator

Accessor Decorator — Same Signature as Method

Applied to a get or set accessor. Same signature as method decorators — (target, key, descriptor) — but here descriptor.get/descriptor.set matter instead of descriptor.value.

Direct form — @ReadOnly

function ReadOnly(target: any, key: string, desc: PropertyDescriptor) {
  desc.set = function() {
    throw new Error(`${key} is read-only`);
  };
}

class User {
  private _role = "admin";

  @ReadOnly
  get role() { return this._role; }
}
// new User().role = "x"  → ❌ Error: role is read-only

Factory form — @Validate(fn)

function Validate(check: (v: any) => boolean) {
  return function(target: any, key: string, desc: PropertyDescriptor) {
    const originalSet = desc.set!;
    desc.set = function(val: any) {
      if (!check(val)) throw new Error(`Invalid ${key}`);
      originalSet.call(this, val);
    };
  };
}
Use Cases

Common patterns: enforce read-only properties at runtime, validate values before setting, lazy-load on first get access, caching computed values.

Property Decorator

Property Decorator — (target, propertyKey) => void

Receives only target (prototype) and propertyKey (name). No descriptor — you cannot directly modify the value. Instead, you collect metadata or register the property for later use (validation, serialization).

Direct form — @Required

const requiredProps = new Map<string, string[]>();

function Required(target: any, key: string) {
  const name = target.constructor.name;
  const props = requiredProps.get(name) || [];
  props.push(key);
  requiredProps.set(name, props);
}

class Form {
  @Required
  email!: string;
  nickname?: string;
}
// requiredProps.get("Form") → ["email"]

Factory form — @Column("varchar")

function Column(type: string) {
  return function(target: any, key: string) {
    // store metadata: { class: "User", property: "name", dbType: "varchar" }
    Reflect.defineMetadata("db:type", type, target, key);
  };
}

Why no descriptor? — TypeScript calls property decorators at class definition, but the property value only exists after instantiation. So decorators here only register metadata — the actual work happens later (in a validator, serializer, or ORM query builder).

Use Cases

TypeORM: @Column(), @PrimaryGeneratedColumn() · class-validator: @IsEmail(), @MinLength(5) · Common patterns: marking required fields, defining DB schema, JSON serialization rules, dependency injection tokens.

Parameter Decorator

Parameter Decorator — (target, methodName, paramIndex) => void

Receives target (prototype), methodName (which method the param belongs to), and parameterIndex (0-based position). Like property decorators, it only registers metadata — it cannot modify the value.

Direct form — @Inject

function Inject(target: any, method: string, index: number) {
  console.log(`Inject param[${index}] in ${method}`);
}

class Service {
  save(@Inject data: string) { /* … */ }
}
// → "Inject param[0] in save"

Factory form — @Label("user data")

function Label(name: string) {
  return function(target: any, method: string, index: number) {
    console.log(`param[${index}] in ${method} → "${name}"`);
  };
}

class Service {
  save(@Label("user data") data: string) { /* … */ }
}
// → "param[0] in save → "user data""

Why combine with a method decorator? — Parameter decorators alone can only store metadata. Pair them with a method decorator that reads that metadata and validates arguments at runtime. This is how NestJS @Body(), @Param(), and @Query() work — the parameter decorator marks which arg to inject, the method/route decorator does the actual injection.

Use Cases

NestJS: @Body(), @Param("id"), @Query() · Angular: @Inject(TOKEN) in constructor · Common patterns: dependency injection, parameter validation metadata, marking params for serialization/transformation.

Basics

Generic Function — Preserve the Type

Definition: Generics allow you to write reusable, type-safe code that works across many types rather than a single one. Instead of using any (which loses type information), generics preserve the type relationship between inputs and outputs.

Generics vs any: any loses the type. Generics preserve it — the caller and return type stay linked. Key pattern: K extends keyof T ensures you can only pass valid property names of T.

function identity<T>(arg: T): T { return arg; }

identity<string>("hello"); // → "hello" (type: string)
identity(42);              // → 42     (type: number — inferred)
identity(true);            // → true   (type: boolean)
Interview Tip

Say: "Generics are type variables — they let a function work with many types while keeping them connected, unlike any which erases the link."

With type

Generic with type Alias

A generic type alias creates reusable type shapes — wrap any value, define result patterns, compose types.

type Box<T> = { value: T };
type Result<T> = { ok: true; data: T } | { ok: false; error: string };

const numBox: Box<number> = { value: 42 };    // → { value: 42 }
const strBox: Box<string> = { value: "hi" }; // → { value: "hi" }

const pass: Result<number> = { ok: true, data: 10 };
const fail: Result<number> = { ok: false, error: "timeout" };
With extends

Generic Constraint — T extends

Restrict what T can be. Without a constraint, you can't access properties on T.

function getLength<T extends { length: number }>(arg: T): number {
  return arg.length;
}

getLength("hello");     // → 5   (string has .length)
getLength([1,2,3]);    // → 3   (array has .length)
// getLength(42);       // ❌ number has no .length

// keyof constraint — only valid keys allowed
function pick<T, K extends keyof T>(obj: T, key: K): T[K] {
  return obj[key];
}

pick({ name: "Ani", age: 23 }, "name");  // → "Ani" (type: string)
pick({ name: "Ani", age: 23 }, "age");   // → 23   (type: number)
// pick({ name: "Ani" }, "foo");           // ❌ "foo" not in keyof
With interface & implements

Generic Interface + Class implements

Define a generic contract, then implement it with a concrete type.

interface Repository<T> {
  getById(id: number): T;
  save(item: T): void;
}

class UserRepo implements Repository<{ name: string }> {
  getById(id: number) { return { name: "User" + id }; }
  save(item: { name: string }) { console.log("saved", item.name); }
}

const repo = new UserRepo();
repo.getById(1);           // → { name: "User1" }
repo.save({ name: "Ani" }); // → "saved Ani"

Generic Class — Reusable Data Structures

A class with a type parameter — the type is fixed when you instantiate.

class Stack<T> {

const nums = new Stack<number>();

With Tuples & Enums

Generics with Tuples & Enum-like Patterns

Generics compose naturally with tuples and enum-style constants.

// Generic tuple — swap positions
function swap<A, B>(pair: [A, B]): [B, A] {
  return [pair[1], pair[0]];
}

swap(["hello", 42]);  // → [42, "hello"]  type: [number, string]
swap([true, "x"]);   // → ["x", true]   type: [string, boolean]

// Generic constrained to enum-like union
const Status = { Active: "active", Inactive: "inactive" } as const;
type StatusVal = typeof Status[keyof typeof Status];

function setStatus<T extends StatusVal>(s: T): T { return s; }

setStatus("active");    // → "active"   type: "active"
setStatus("inactive");  // → "inactive" type: "inactive"
// setStatus("deleted"); // ❌ not in StatusVal

// Generic that returns a typed tuple from an object
function entries<T>(obj: T): [keyof T, T[keyof T]][] {
  return Object.entries(obj) as any;
}

entries({ x: 1, y: 2 });  // → [["x",1],["y",2]]  type: ["x"|"y", number][]
Defaults & Multi-Param

Default Type Parameters & Multiple Generics

Like function defaults — provide a fallback type when the caller doesn't specify one.

type ApiResponse<T = unknown> = { data: T; status: number };

const r1: ApiResponse<string> = { data: "ok", status: 200 };
const r2: ApiResponse = { data: 123, status: 200 }; // T defaults to unknown

// Multiple params — map one type to another
function map<In, Out>(arr: In[], fn: (x: In) => Out): Out[] {
  return arr.map(fn);
}

map([1,2,3], n => n * 2);         // → [2,4,6]           type: number[]
map(["a","b"], s => s.length); // → [1,1]             type: number[]
map([1,2], n => String(n));     // → ["1","2"]         type: string[]
Interview Tip

Say: "Generics vs any — any erases the type, generics preserve it. The input and output stay linked, so the caller always gets back the exact type they put in."

Overview

extends — One Keyword, Many Meanings

The extends keyword appears in 5 different contexts in TypeScript. Each means something slightly different, but the core idea is always: "must be compatible with" or "builds upon".

ContextMeaningExample
Class inheritance Subclass inherits from parent class Dog extends Animal
Interface inheritance Interface builds on another interface Admin extends User
Generic constraint T must be assignable to X T extends { id: number }
Conditional type Type-level if/else check T extends string ? A : B
Type narrowing (infer) Extract inner type T extends Array<infer U> ? U : T
Class extends

Class Inheritance — class Child extends Parent

The child class gets all properties and methods from the parent. It can override methods and add new ones. Only single inheritance — one parent class only.

class Animal {
  constructor(public name: string) {}
  speak() { return `${this.name} makes a sound`; }
}

class Dog extends Animal {
  speak() { return `${this.name} barks`; }  // override
  fetch() { return "fetches ball"; }     // new method
}

const d = new Dog("Rex");
d.speak();  // → "Rex barks"
d.fetch();  // → "fetches ball"
d.name;    // → "Rex" (inherited from Animal)
d instanceof Animal;  // → true
Rule

Child constructor must call super() before accessing this. A class can only extend one parent — for multiple contracts, use implements with interfaces.

Interface extends

Interface Inheritance — interface B extends A

An interface can extend one or multiple other interfaces (or even type aliases). All members are merged into the child.

interface HasId   { id: number; }
interface HasName { name: string; }

interface User extends HasId, HasName {
  email: string;
}

const user: User = { id: 1, name: "Ani", email: "a@b.com" };
// Must have all 3: id + name + email

// Can also extend a type alias
type Timestamps = { createdAt: Date; updatedAt: Date; };
interface Post extends Timestamps {
  title: string;
}

const post: Post = { title: "Hi", createdAt: new Date(), updatedAt: new Date() };
Key Difference

Unlike class extends (single parent only), interface extends supports multiple parents. Conflicts in property types are caught immediately at the extends declaration.

Generic constraint

Generic Constraint — T extends Shape

Restricts what types can be passed as T. Without it, T could be anything and you can't access any properties.

// T must have a .length property
function logLength<T extends { length: number }>(item: T): T {
  console.log(item.length);
  return item;
}

logLength("hello");    // → logs 5, returns "hello"
logLength([1,2,3]);  // → logs 3, returns [1,2,3]
// logLength(42);      // ❌ number has no .length

// T constrained to keys of another type
function pick<T, K extends keyof T>(obj: T, key: K): T[K] {
  return obj[key];
}

pick({ a: 1, b: 2 }, "a");  // → 1 (type: number)
// pick({ a: 1 }, "x");       // ❌ "x" not in "a"|"b"

// T constrained to an interface
interface Printable { print(): void; }

function doPrint<T extends Printable>(item: T) { item.print(); }
Conditional type

Conditional Types — T extends U ? A : B

A type-level ternary. If T is assignable to U, result is A, otherwise B. This is how utility types like Exclude, Extract, NonNullable work internally.

type IsString<T> = T extends string ? "yes" : "no";

type A = IsString<string>;   // → "yes"
type B = IsString<number>;   // → "no"
type C = IsString<"hello">;  // → "yes" (literal extends string)

// With infer — extract inner type
type Unwrap<T> = T extends Promise<infer U> ? U : T;

type X = Unwrap<Promise<number>>;  // → number
type Y = Unwrap<string>;           // → string (not a Promise, returns T)

// Extract array element type
type Elem<T> = T extends (infer U)[] ? U : never;

type E = Elem<string[]>;  // → string
type F = Elem<number>;    // → never (not an array)
Interview Tip

Here extends means "is assignable to" — not inheritance. It's a type-level boolean check. Combined with infer, it lets you pattern-match on types and pull out pieces.

Summary

Quick Reference — All 5 Uses of extends

// 1. Class inherits from parent
class Cat extends Animal {}

// 2. Interface builds on others
interface Admin extends User, HasPermissions {}

// 3. Generic constraint — T must have .id
function save<T extends { id: number }>(entity: T) {}

// 4. Conditional — type-level if/else
type Check<T> = T extends null ? "null" : "value";

// 5. infer — extract inner type
type ReturnOf<T> = T extends (...args: any) => infer R ? R : never;

// Usage of ReturnOf
type R = ReturnOf<() => string>;  // → string
type S = ReturnOf<number>;        // → never
Interview Tip

When asked "what does extends mean?" — answer: it depends on context. In classes/interfaces it means inheritance. In generics it means constraint. In conditional types it means "is assignable to" — a type-level boolean test.

Advanced

Enums — Numeric

Rule: Numeric enums auto-increment from 0 by default. If you set any member to a number, the next member increments from that value — not from 0. Unset members before the first set member default to 0, 1, 2…

enum Direction { Up, Down, Left, Right }
// Up=0  Down=1  Left=2  Right=3   (default: starts at 0)

enum FileAccess {
  None,          // 0  (starts at 0)
  Read  = 1 << 1, // 2  (explicit)
  Write = 1 << 2, // 4  (explicit)
  ReadWrite = Read | Write, // 6  (computed)
  Special = 99,  // 99 (explicit reset)
  Next,          // 100 — increments from PREVIOUS (99)
}
console.log(FileAccess.Next); // 100, not 4!

// ⚠️ Reverse mapping — numeric only
enum E { A = 5, B, C }
console.log(E.B);    // 6   (forward)
console.log(E[6]);   // "B" (reverse mapping!)
// The compiled JS object has BOTH E[5]="A" AND E["A"]=5
Watch Out

Auto-increment always continues from the previous member's value, not from 0. Setting X = 99 makes the next unset member 100. This makes inserting members in the middle silently break all following numeric values.

Advanced

Enums — String

Rule: String enums have no auto-increment. Every member after the first one that has a string value must be explicitly assigned — otherwise TypeScript errors. String enums don't get a reverse mapping.

enum Status {
  Active   = "ACTIVE",   // ✅ explicit
  Inactive = "INACTIVE", // ✅ explicit — required
  Pending  = "PENDING",  // ✅ explicit — required
  // Draft,              ❌ Error: no auto-increment for strings
}

// No reverse mapping for string enums:
console.log(Status["ACTIVE"]); // undefined (unlike numeric)

// Mixed enum (legal but confusing — avoid)
enum Mixed {
  A,           // 0 — numeric, gets reverse map
  B = "hello", // string — no reverse map
  C = 10,      // back to numeric
  D,           // 11 — increments from 10
}

// Why string enums are better for logs/debugging:
// console.log(Status.Active) → "ACTIVE"  (readable)
// console.log(Direction.Up) → 0          (just a number)
Interview Tip

Prefer string enums over numeric for any value that crosses a network boundary (API, logs, storage) — you get readable values instead of opaque integers. But even string enums emit a JS object at runtime — see const enum and as const for zero-cost alternatives.

Advanced

Enums — const enum

Definition: A const enum is fully erased at compile time — every usage is inlined as the literal value. No JavaScript object is emitted at all. This is the most performant form, but has critical restrictions.

const enum Direction { Up = "UP", Down = "DOWN" }

// TypeScript source:
let d = Direction.Up;
if (d === Direction.Down) { /* ... */ }

// ↓ Compiled JavaScript output:
let d = "UP";            // inlined — no Direction object exists
if (d === "DOWN") { }   // inlined

// ⚠️ CANNOT do runtime lookup — object doesn't exist:
// Direction["UP"]  → ReferenceError at runtime
// const key = "Up"; Direction[key] → also broken

// ⚠️ Breaks with isolatedModules (Vite / esbuild / Babel)
// because each file is compiled independently — inlining
// across files is impossible without the full program.

// ⚠️ Breaks with declaration files (.d.ts) from external libs
// — you can't inline what you can't see at compile time.

Use const enum when

  • You use tsc directly (not Babel/esbuild)
  • No runtime lookup is ever needed
  • Values are all in one project (not a lib)

Avoid const enum when

  • Using Vite, Next.js, esbuild, or Babel
  • Publishing a library (.d.ts consumers break)
  • You need isolatedModules: true
Advanced

Enums — as const & Union Literals (Preferred)

Modern TypeScript teams use as const objects or plain union literals instead of enum. They are tree-shakeable, work with every bundler, and produce the same type safety.

Pattern 1 — Plain union literal (simplest)

// No object at all — pure type, zero runtime cost
type Role = "user" | "admin" | "superadmin";

function redirect(role: Role) { /* ... */ }
redirect("admin");   // ✅
// redirect("god"); // ❌ Error

Pattern 2 — as const object (named values + union type)

const Role = {
  User:       "user",
  Admin:      "admin",
  SuperAdmin: "superadmin",
} as const;

// Derive the union type from values:
type Role = typeof Role[keyof typeof Role];
// → "user" | "admin" | "superadmin"

redirect(Role.Admin);  // ✅ readable named constant
redirect("admin");     // ✅ literal also accepted

// Derive keys if you need them:
type RoleKey = keyof typeof Role; // "User" | "Admin" | "SuperAdmin"

Pattern 3 — Nested group with as const (replaces enum namespacing)

// "role" | ("user" | "admin") — build const values under a namespace
const Permission = {
  role: {
    user:  "role:user",
    admin: "role:admin",
  },
  action: {
    read:   "action:read",
    write:  "action:write",
    delete: "action:delete",
  },
} as const;

type PermissionValue =
  typeof Permission[keyof typeof Permission][
    keyof typeof Permission[keyof typeof Permission]
  ];
// → "role:user" | "role:admin" | "action:read" | "action:write" | "action:delete"

console.log(Permission.role.admin); // "role:admin"

Full comparison

enum

enum Role {
  User  = "user",
  Admin = "admin",
}
// Emits JS object at runtime
// NOT tree-shakeable
// Breaks isolatedModules (const enum)
// String enum: no reverse map
// Numeric enum: has reverse map

as const

const Role = {
  User:  "user",
  Admin: "admin",
} as const;

// Tree-shakeable ✅
// Works everywhere ✅
// No reverse map (not needed)
// Literal type union derived
Interview Tip

The canonical answer: prefer as const + derived union type over enum for three reasons: ① tree-shakeable, ② works with isolatedModules (Vite/esbuild), ③ no surprise reverse mapping. Only reach for enum when you're in a pure-tsc project and need the runtime object by name.

Fundamentals

What is a Tuple? — Array vs Tuple

A tuple is a fixed-length array where every position has its own specific type. Use an array when all items have the same type and count is variable. Use a tuple when item count is fixed and each position means something different.

Array — same type, any length 
// All strings, could be 0–N items
const names: string[] = ["Alice", "Bob"];

// TypeScript can't tell you what
// names[0] "means" — it's just a string
const first = names[0]; // string
Tuple — typed positions, fixed length 
// Position 0 = id, position 1 = label
type Entry = [number, string];
const row: Entry = [1, "Alice"];

const id  = row[0]; // number ✅
const lbl = row[1]; // string ✅

// ❌ Wrong order — TypeScript catches it
const bad: Entry = ["Alice", 1];
// Type 'string' is not assignable to type 'number'

// Named tuple elements (TS 4.0+) — labels show in IDE tooltips
type Point = [x: number, y: number];
const p: Point = [10, 20];
// Hovering p[0] shows "x: number" instead of just "number"

// Destructuring — the cleanest way to consume a tuple
const [x, y] = p;   // x: number, y: number
Interview Tip

"When should you use a tuple vs an object?" — Object when the fields have meaningful names you'll refer to by name. Tuple when the structure is positional and temporary (e.g. a return value you'll immediately destructure).

Use Case 1

Returning Multiple Values from a Function

The most common real-world use of tuples. Instead of returning an object (verbose) or an array (loses type info), return a typed tuple that destructures cleanly at the call site.

// ── Pattern 1: [data, error] result pair ──────────────
type Result<T> = [data: T | null, error: string | null];

async function fetchUser(id: number): Promise<Result<User>> {
  try {
    const user = await getUser(id);
    return [user, null];       // success
  } catch (e) {
    return [null, "Not found"];  // failure
  }
}

// Call site — destructure and handle both cases
const [user, err] = await fetchUser(1);
// user: User | null,  err: string | null
if (err) return console.error(err);
console.log(user.name); // safe — narrowed to User

// ── Pattern 2: React-style [value, setter] hook ───────
function useToggle(init: boolean): [boolean, () => void] {
  let val = init;
  const toggle = () => { val = !val; };
  return [val, toggle];
}

const [isOpen, toggleOpen] = useToggle(false);
// isOpen: boolean,  toggleOpen: () => void

// ── Pattern 3: [min, max] bounds pair ─────────────────
function getRange(nums: number[]): [min: number, max: number] {
  return [Math.min(...nums), Math.max(...nums)];
}
const [min, max] = getRange([3, 1, 9, 2]); // [1, 9]
Gotcha

Always annotate the return type explicitly: : [boolean, () => void]. Without it TypeScript infers (boolean | (() => void))[] — a union array — and you lose per-position types.

Use Case 2

Tuple as Function Parameters — Spread & Apply

A tuple can represent an exact argument list for a function. Spreading a tuple into a call site is the type-safe replacement for Function.prototype.apply.

// ── Spread tuple as arguments ─────────────────────────
function add(a: number, b: number): number {
  return a + b;
}

const args: [number, number] = [3, 7];
add(...args); // ✅  — TypeScript knows the spread is exactly 2 numbers

// Without a tuple TypeScript complains:
const loose = [3, 7];            // inferred as number[]
// add(...loose);  // ❌ spread of number[] not assignable to [number,number]

// ── Rest parameters typed as a tuple ──────────────────
type LogArgs = [level: string, message: string, ...unknown[]];

function log(...args: LogArgs) {
  const [level, message, ...extra] = args;
  // level: string, message: string, extra: unknown[]
  console.log(`[${level}]`, message, ...extra);
}
log("INFO", "Server started", { port: 3000 });

// ── Generic "apply" wrapper ────────────────────────────
function call<T extends unknown[], R>(
  fn: (...args: T) => R,
  args: T
): R {
  return fn(...args);
}
call(add, [2, 5]);  // T=[number,number], R=number ✅
Interview Tip

Parameters<typeof fn> extracts the argument tuple of any function. Use it to build wrapper types without repeating parameter lists: type AddArgs = Parameters<typeof add>; // [number, number]

Use Case 3

Optional & Rest Elements — Flexible Tuple Shapes

Tuples can have optional trailing elements (?) and variadic rest elements (...T[]), making them useful for modelling APIs with optional overloads.

// ── Optional element ──────────────────────────────────
type Route = [path: string, statusCode?: number];

function redirect(...args: Route) {
  const [path, code = 302] = args;
  console.log(`Redirect to ${path} with ${code}`);
}
redirect("/home");          // ✅ uses default 302
redirect("/login", 301);   // ✅ permanent redirect

// ── Rest element — variadic tail ──────────────────────
type EventPayload = [eventName: string, timestamp: number, ...unknown[]];

function emit(...payload: EventPayload) {
  const [name, ts, ...data] = payload;
  // data can hold any extra event-specific values
}
emit("click", Date.now(), { x: 100, y: 200 });

// ── Rest at the start (TS 4.2+) ───────────────────────
type TrailingId = [...string[], number];
// all elements except the last are strings; last is always a number
const row: TrailingId = ["a", "b", "c", 42];
Gotcha

Only one rest element per tuple, and optional elements must come after all required ones. [string?, number] is a compile error — optional can't precede required.

Use Case 4

Readonly Tuples & as const — Immutable Data

Make a tuple immutable with readonly (allows any number, freezes mutability) or as const (freezes both mutability and values to literals).

// ── readonly keyword ──────────────────────────────────
type Coord = readonly [number, number];
const origin: Coord = [0, 0];
// origin[0] = 1;  // ❌ Index signature in type 'readonly [...]' only permits reading

// ── as const — literal + readonly ─────────────────────
const ORIGIN = [0, 0] as const;
// type: readonly [0, 0]  — values are literal 0, not just number

// Use case: fixed config tuples
const RGB_RED = [255, 0, 0] as const;
// type: readonly [255, 0, 0]

// ── The mismatch gotcha ───────────────────────────────
function move([x, y]: [number, number]) { /* ... */ }

// move(origin);   // ❌ readonly not assignable to mutable
move([...origin]);      // ✅ spread makes a fresh mutable copy
move(origin as [number, number]); // ✅ assertion (use sparingly)

// ── Better: accept readonly in the function ───────────
function distance([x, y]: readonly [number, number]) {
  return Math.sqrt(x ** 2 + y ** 2);
}
distance(ORIGIN); // ✅ readonly accepted, no spread needed
Interview Tip

Design functions to accept readonly tuples when you don't need to mutate them — this makes them work with both mutable and immutable callers. The rule: be liberal in what you accept.

Use Case 5

Real-World Tuple Patterns

Five practical patterns that appear frequently in production TypeScript code.

// ── 1. CSV / table row ────────────────────────────────
type UserRow = [id: number, name: string, active: boolean];
const rows: UserRow[] = [
  [1, "Alice", true],
  [2, "Bob",   false],
];
for (const [id, name, active] of rows) { /* typed ✅ */ }

// ── 2. Entry pairs from Object.entries ────────────────
const config = { host: "localhost", port: 3000 };
for (const [key, value] of Object.entries(config)) {
  // key: string, value: string | number
}

// ── 3. Swap two values ────────────────────────────────
function swap<A, B>([a, b]: [A, B]): [B, A] {
  return [b, a];
}
const [num, str] = swap(["hello", 42]);
// num: number, str: string ✅

// ── 4. Zip two arrays into tuple pairs ────────────────
function zip<A, B>(a: A[], b: B[]): [A, B][] {
  return a.map((item, i) => [item, b[i]]);
}
const zipped = zip([1, 2], ["a", "b"]);
// [number, string][]  →  [[1,"a"],[2,"b"]]

// ── 5. Map + filter preserving tuple structure ────────
const scores: [string, number][] = [
  ["Alice", 92], ["Bob", 45], ["Carol", 78],
];
const passed = scores.filter(([, score]) => score >= 60);
// [string, number][] — only Alice and Carol
Interview Tip

Pattern 3 (swap) and 4 (zip) are great interview examples — they show you understand both generic tuples and destructuring. Mention that Object.entries() returns [string, V][] — effectively an array of tuples.

Advanced

Union Types

Definition: A union type describes a value that can be one of several types. Written with the | operator, it tells TypeScript "this can be A or B (or C...)". TypeScript enforces that you handle all possible variants before accessing type-specific properties.

// Basic union
type StringOrNumber = string | number;

// Literal union — restricts to specific values
type Direction = "north" | "south" | "east" | "west";

// Discriminated union — powerful pattern
type Shape =
  | { kind: "circle";    radius: number }
  | { kind: "rectangle"; width: number; height: number };

function area(s: Shape): number {
  switch (s.kind) {
    case "circle":    return Math.PI * s.radius ** 2;
    case "rectangle": return s.width * s.height;
  }
}
Interview Tip

The discriminated union pattern (using a shared literal field like kind as the discriminant) is the most powerful TypeScript pattern. TypeScript can narrow exhaustively and even warn you if you forget a case.

Advanced

Intersection Types

Definition: An intersection type combines multiple types into one using &. The resulting type has all properties of every constituent type — the value must satisfy every type simultaneously. Think of it as "A AND B".

type Serializable = { serialize(): string };
type Printable    = { print(): void };

// Intersection — must have BOTH methods
type Document = Serializable & Printable;

const doc: Document = {
  serialize() { return "{}"; },
  print() { console.log(this.serialize()); }
};

// Common use: merging props with metadata
type WithId<T> = T & { id: string; createdAt: Date };

// Impossible intersection — results in never
type Impossible = string & number; // type: never
Interview Tip

Union vs Intersection: Union = "OR" (either A or B). Intersection = "AND" (both A and B). When two incompatible primitives are intersected (like string & number), the result is never.

Advanced

Type Guards

Definition: A type guard is a runtime check that narrows a type within a conditional block. TypeScript recognises typeof, instanceof, in, and user-defined predicates using the is keyword.

// typeof guard (primitives)
function process(val: string | number) {
  if (typeof val === "string") {
    return val.toUpperCase();
  }
  return val.toFixed(2);
}

// instanceof guard (classes)
if (err instanceof NetworkError) {
  retry(err.endpoint);
}

// in guard (property existence)
if ("radius" in shape) {
  return Math.PI * shape.radius ** 2;
}

// User-defined type guard (predicate)
function isString(val: unknown): val is string {
  return typeof val === "string";
}
if (isString(input)) {
  input.toLowerCase(); // TS knows: input is string
}
Interview Tip

User-defined type guards (x is Type) are the key advanced concept. Without the is predicate, TypeScript can't carry the narrowing out of the function. Critical for validating unknown values from API responses or JSON.parse.

Advanced

Narrowing

Definition: Narrowing is the process by which TypeScript refines a broader type to a more specific one based on control flow analysis. TypeScript tracks every branch and knows the type at each point.

// Truthiness narrowing
function printName(name?: string) {
  if (name) {
    console.log(name.toUpperCase()); // string here
  }
}

// Equality narrowing
function compare(a: string | number, b: string | boolean) {
  if (a === b) {
    a.toUpperCase(); // both must be string here
  }
}

// Discriminant narrowing
type Result = { ok: true;  value: number }
            | { ok: false; error: string };

function handle(r: Result) {
  if (r.ok) { return r.value; }
  return r.error;
}

// Exhaustiveness check with never
function assertNever(x: never): never {
  throw new Error("Unhandled case: " + x);
}
Interview Tip

The exhaustiveness check with never is a key interview topic. When you add a new variant to a union, TypeScript will error at the assertNever call — forcing you to handle it. This makes union types safe to evolve over time.

Overview

Special Types — any · unknown · never · void

Four types that don't represent normal values — they model escape hatches, safety boundaries, impossibility, and absence of return.

TypeMeaningCan assign TO itCan assign FROM it
any Escape hatch — disables checking ✔ anything ✔ anything
unknown Safe top type — must narrow before use ✔ anything ✘ only unknown or any
never Bottom type — impossible / unreachable ✘ nothing (only never) ✔ everything
void Function returns nothing useful ⚠ only undefined ✘ not assignable to other types
Interview Tip

The type hierarchy: never is at the bottom (subtype of everything), unknown is at the top (supertype of everything), any breaks the hierarchy entirely — it's both top AND bottom simultaneously.

Dangerous

any — The Type System Escape Hatch

Disables all type checking. A value of type any can be assigned to anything and anything can be assigned to it. Use only when migrating JS or when types are genuinely unknowable.

let x: any = "hello";
x = 42;              // ✅ no error
x.nonExistent();     // ✅ no error — compiles, crashes at runtime

const n: number = x; // ✅ any flows into number silently

Union with any:

type A = string | any;    // → any  (any absorbs everything)
type B = string & any;    // → any  (any wins intersections too)

Interview trap:

function add(a: number, b: number) { return a + b; }
const val: any = "oops";
add(val, 1);  // ✅ compiles — "oops1" at runtime (silent bug!)
Rule

any is contagious — once it enters an expression, everything downstream loses type safety. Prefer unknown and narrow instead.

Safe Alternative

unknown — The Safe Top Type

Like any, you can assign anything TO it. Unlike any, you cannot use it until you narrow it. Forces you to check before accessing.

let x: unknown = "hello";
x = 42;               // ✅ assigning anything to unknown is fine

// x.toUpperCase();   // ❌ cannot access without narrowing
// const n: number = x; // ❌ cannot assign unknown to number

  x.toUpperCase();   // ✅ narrowed to string

Union / Intersection with unknown:

type A = string | unknown;   // → unknown (top type absorbs in union)
type B = string & unknown;   // → string  (intersect with top = no change)

Real use case — safe JSON parsing:

  return JSON.parse(input);
}

Interview Tip

"When should I use unknown over any?" — Always. unknown gives you the same flexibility for receiving values, but forces you to prove the type before using it. Use any only at JS/TS migration boundaries.

Bottom Type

never — The Impossible Type

never means a value can never exist. A function that always throws, an infinite loop, or exhausted union branches. It's the bottom of the type hierarchy — assignable TO everything, nothing is assignable to it.

// Function that never returns
function throwError(msg: string): never {
  throw new Error(msg);
}

// Exhaustiveness check — the classic pattern
type Shape = "circle" | "square";
function area(s: Shape) {
  switch(s) {
    case "circle": return 3.14;
    case "square": return 4;
    default: const _: never = s; // ❌ if new variant added
  }
}

Union / Intersection with never:

type A = string | never;   // → string (never disappears from unions)
type B = string & never;   // → never  (intersect with never = never)

Interview edge case — conflicting intersection:

type X = { id: number } & { id: string };
// X.id is type: never (number & string = impossible)
// You can declare X but can never assign a value to it
Interview Tip

never is TypeScript's way of saying "this should be unreachable." If you see it unexpectedly, it means your types have conflicting constraints. The exhaustiveness pattern (const _: never = x) is a must-know for interviews.

Return Type

void — Function Returns Nothing Useful

void means "this function doesn't return a useful value." It's not the same as undefined — there's a subtle but important difference in callback types.

// Basic — function returns nothing
function log(msg: string): void {
  console.log(msg);
  // no return statement
}

The surprising rule — void in callback types:

// A callback typed as returning void CAN return a value
type Callback = () => void;
const fn: Callback = () => 42;  // ✅ no error!

// Why? So Array.forEach works:
[1,2].forEach(n => n * 2); // ✅ return value just ignored

void vs undefined vs never:

function a(): void      {}       // returns undefined implicitly — OK
function b(): undefined { return; } // must explicitly return undefined
function c(): never     { throw "!"; } // never returns at all

Union / Intersection with void:

type A = string | void;   // → string | void  (void stays, value could be undefined)
type B = string & void;   // → never  (string & void = impossible)
Gotcha

void in a function declaration means "must not return a value." But void in a function type (callback) means "return value will be ignored" — it doesn't prevent returning. This is intentional design for compatibility with forEach, map callbacks.

Bonus

Rapid-Fire Q&A — 10 Quick Answers

Click each question to reveal the answer. Aim to answer in your head before opening.

Q1 — What is the difference between unknown and any?

any disables all type checking — you can call methods, index, assign it anywhere. unknown is the type-safe top type — you can assign anything to it, but must narrow before use. Prefer unknown for values from external sources.

Q2 — When does TypeScript produce a never type?

Three cases: (1) a function that never returns (throw or infinite loop), (2) an impossible intersection like string & number, (3) the remaining type after exhaustive narrowing in a union — used for exhaustiveness checks.

Q3 — What is structural typing and how does it differ from nominal typing?

TypeScript uses structural typing: two types are compatible if they have the same shape, regardless of name. Nominal typing (e.g. Java/C#) requires the same declared class/interface name. In TS, { name: string } satisfies interface User { name: string } even without implements.

Q4 — What is declaration merging and which constructs support it?

Declaration merging means TypeScript automatically combines multiple declarations with the same name. Only interfaces, namespaces, and enums support it. type aliases do not — re-declaring a type alias is a compile error.

Q5 — What is the difference between type and interface for extending?

interface uses extends (eager, checked at definition time). type uses & intersection (lazy, may produce never on conflict). Key difference: interfaces support declaration merging; types do not.

Q6 — What does keyof return?

keyof T returns a union of all known public property names of T. For { x: number; y: string } it produces "x" | "y". For a type with a [key: string] index signature it returns string | number.

Q7 — What is the infer keyword used for?

infer is used inside conditional types to capture a type variable. Example: type ReturnType<T> = T extends (...args: any[]) => infer R ? R : never extracts R — the return type — from any function type T.

Q8 — What is covariance vs contravariance in TypeScript?

Covariant (out T): subtypes flow outward — Producer<Dog> is assignable to Producer<Animal> if Dog extends Animal. Contravariant (in T): direction reverses for inputs — Consumer<Animal> is assignable to Consumer<Dog>. Functions are covariant in return types and contravariant in parameter types.

Q9 — Why should you use as const instead of enum?

Three reasons: (1) as const objects are tree-shakeable — unused values are eliminated. (2) They work with isolatedModules (Vite/esbuild), unlike const enum. (3) No reverse-mapping overhead at runtime. The derived type gives you the same literal union.

Q10 — What is the excess property check and when does it apply?

When you assign an object literal directly to a typed variable, TypeScript rejects extra properties (excess property check). It does not apply when you assign via an intermediate variable, because that goes through structural compatibility only. Fix: add an index signature, use a type assertion, or assign via a variable.

💡 Hint

✅ Solution 
// Approach 1 — typeof branches

20 Generic Mixin Function with &

Use object spread: { ...a, ...b }. Cast the return as A & B — TypeScript can't infer the spread as an intersection automatically so you need as A & B. The spread ensures b wins on key conflict at runtime.

function mixin<A, B>(a: A, b: B): A & B {
